# 趣学伴 1.8 筹备方案

更新时间：2026-04-27

## 版本定位

`1.8 = UI 精修 + 管理体系升级 + 数据源治理版`

1.5 解决“能上线、主流程顺、页面统一”。  
1.8 要解决“管理体系更专业、页面数据更可信、家长知道下一步怎么管”。

## 1.8 三个核心目标

1. 全站 UI 做微调，不大改结构，但提升一致性、密度、可读性和弹窗质量。
2. 建立孩子 L1-L5 成长分级体系，把任务、阅读、体育、习惯、目标都放进同一套管理框架。
3. 梳理所有页面数据源，确保页面上的每个数字、趋势、建议都有真实数据积累和明确口径。

## 一、UI 微调范围

### 1. 页面顶部

- 所有一级页面统一使用 `PageToolbar` 或同等外观标准。
- 固定宽度标准：以今日概览 `max-w-[1360px]` 为基准。
- 固定内部节奏：图标块、标题、描述、右侧操作。
- 同类按钮统一颜色和高度。

### 2. 卡片与弹窗

- 卡片圆角、边框、阴影统一。
- 弹窗标题、说明、表单间距、底部按钮统一。
- 删除、重置、危险操作弹窗单独建立危险样式。
- 表单错误提示统一位置和文案。

### 3. 数据展示

- 指标卡减少装饰性文案，强化指标名称、口径、变化。
- 趋势图统一时间轴、空状态、加载状态。
- 运营类指标必须有解释入口或 tooltip。

### 4. 移动端

- 顶部孩子切换、通知、菜单不遮挡。
- 关键表格/网格移动端改为卡片或横向滚动。
- 所有弹窗在移动端可滚动、按钮不被遮挡。

## 二、孩子 L1-L5 成长分级体系

### 分级原则

L1-L5 不是年级，也不是单纯年龄，而是家庭管理阶段：

- `L1 启蒙建立期`：建立学习和生活习惯。
- `L2 基础稳定期`：形成固定节奏，提升基础完成质量。
- `L3 能力成长期`：开始关注学科能力、阅读质量、体育指标。
- `L4 自主提升期`：逐步让孩子承担计划、复盘和选择。
- `L5 自驱规划期`：面向长期目标、竞赛/专项、升学规划和自我管理。

### L1 标准建议

- 学科：能稳定完成短时任务，重视兴趣和基础认知。
- 阅读：亲子共读为主，建立每日阅读习惯。
- 体育：每天基础活动，关注协调性、平衡、柔韧。
- 习惯：固定作息、收纳、完成后反馈。
- 家长介入：高介入，任务拆小，反馈及时。

### L2 标准建议

- 学科：语数英基础训练开始稳定，能完成 15-25 分钟任务。
- 阅读：开始自主阅读短篇，能复述主要内容。
- 体育：基础体能、跳绳、跑步、核心动作稳定。
- 习惯：能按计划完成多数任务，需要家长提醒。
- 家长介入：中高介入，重点看稳定性和错因。

### L3 标准建议

- 学科：开始关注正确率、错题类型、知识点薄弱项。
- 阅读：章节阅读、读后表达、阅读类型均衡。
- 体育：形成每周运动计划，记录体能指标变化。
- 习惯：能参与制定计划，完成后做简单复盘。
- 家长介入：中介入，重点做策略调整。

### L4 标准建议

- 学科：能按目标安排练习，关注效率和难度提升。
- 阅读：能做主题阅读、摘录、观点表达。
- 体育：有专项提升目标，关注周期训练和恢复。
- 习惯：能独立执行大部分计划，主动汇报问题。
- 家长介入：低中介入，重点做阶段复盘。

### L5 标准建议

- 学科：围绕长期目标做规划，支持专项和竞赛。
- 阅读：形成个人阅读结构和表达产出。
- 体育：稳定训练和健康管理，支持专项目标。
- 习惯：自主管理、主动复盘、长期计划能力。
- 家长介入：低介入，系统更多提供趋势和建议。

## 三、L1-L5 与系统模块关系

### 任务管理

- 每个任务增加推荐阶段：L1-L5。
- 每个任务增加能力标签：学科、阅读、体育、习惯、表达、自主性。
- 每个任务增加难度和时长建议。

### 学习计划

- 计划生成时参考当前孩子阶段。
- 不同阶段默认任务数量、单任务时长、家长参与程度不同。
- 计划页显示“是否适合当前阶段”的提示。

### 今日概览

- 不只显示完成情况，还显示阶段目标执行情况。
- 例如：L2 重点看稳定完成，L4 重点看自主完成和复盘。

### 仪表盘

- 按 L1-L5 展示当前孩子能力画像。
- 每个维度显示当前等级、趋势、风险、下一步建议。

### 目标管理

- 目标模板内置分级标准。
- 家长可以选择当前阶段目标，也可以自定义。
- 目标进度来自任务、阅读、体育、习惯数据。

## 四、页面数据源治理

### 必须建立的数据字典

每个页面字段都要明确：

- 字段名称
- 来源表/接口
- 统计周期
- 是否按孩子隔离
- 是否支持历史回溯
- 空数据如何显示
- 是否能被 AI 使用

### 重点页面数据源

| 页面 | 需要梳理的数据 |
| --- | --- |
| 今日概览 | 今日任务、完成率、学习时长、阅读数量、快捷操作、AI 建议来源 |
| 仪表盘 | 学科能力、阅读能力、体育/习惯、自主性、趋势、风险 |
| 目标管理 | 目标模板、目标进度、阶段标准、建议来源 |
| 任务管理 | 任务频率、完成质量、延期、适用孩子、阶段标签 |
| 学习计划 | 周计划、日期分配、计划变更、完成回流 |
| 图书馆 | 书目、状态、阅读进度、阅读记录、阅读偏好 |
| 学习报告 | 周/月数据聚合、亮点、问题、建议 |
| 设置 | 孩子阶段、家庭偏好、通知配置、数据管理 |

## 五、1.8 里还应补充的事项

- 建立指标口径文档，避免“完成率”“学习时长”“阅读量”在不同页面计算不一致。
- 增加数据质量检测：缺失孩子、无计划、有任务无记录、有图书无状态等。
- 增加变更审计：计划修改、任务删除、图书删除、孩子删除。
- 为 AI 准备事件日志：任务创建、计划发布、完成、调整、删除、阅读更新。
- 建立 UI 检查清单：一级页面、弹窗、空状态、错误状态、移动端。
- 建立多应用服务器部署规范，避免继续把 `/home/ubuntu` 当作单应用 Git 仓库根目录。

### 多应用服务器治理

当前判断：

- 1.7 已经修复单应用部署脚本，但服务器未来会承载多个应用。
- 继续把应用仓库放在 `/home/ubuntu` 会让 `.ssh/`、备份、dump、临时脚本、多个应用文件混进同一个 Git 工作区。
- quxueban 迁移目录会影响 PM2、nginx、上传目录、备份和环境变量，不能夹在 1.7 热修后立即执行。

1.8 目标：

- 制定服务器目录规范，例如 `/home/ubuntu/apps/<app-name>/repo` 或 `/opt/apps/<app-name>/current`。
- 制定端口规范，例如 quxueban 使用 `3001`，后续应用依次使用独立端口。
- 制定 PM2 命名规范，例如 `quxueban-api`、`app2-api`，避免进程名冲突。
- 制定 nginx 入口规范，优先使用域名/子域名区分应用，避免前端 SPA 路径部署复杂化。
- 每个应用独立 `.env`、上传目录、备份目录和部署脚本。
- 新应用从一开始使用新目录规范；quxueban 迁移单独安排维护窗口。

quxueban 迁移验收：

- 新目录 clone、安装、构建通过。
- 在临时端口验证 `/api/health`、登录、图书馆、任务、今日概览。
- nginx 切换后外网资源和接口正常。
- PM2 只保留一个 quxueban 后端进程。
- 老 `/home/ubuntu` 仓库保留一段观察期，不立即删除生产资料。

## 1.8 推荐里程碑

### M1：UI 统一微调

- 顶部长条、按钮、弹窗、卡片统一。
- 移动端检查。

### M2：L1-L5 标准落地

- 先做标准文档和数据结构。
- 再接入任务、目标、计划。

### M3：数据源治理

- 建立数据字典。
- 修正页面指标口径。
- 为所有展示补空状态和来源说明。

### M4：为 2.0 AI 打基础

- 事件日志、指标快照、报告素材沉淀。
- 明确哪些数据可以进入 AI 分析。

### M5：服务器多应用治理

- 先输出多应用目录、端口、PM2、nginx、备份规范。
- 新应用按规范落位。
- quxueban 迁移只在维护窗口执行，迁移前保留完整回滚方案。
