# 趣学伴 1.7-1.9 迭代计划

更新时间：2026-04-28

## 背景

1.6.2 已经完成家庭学习管理主流程的初步闭环：

- 设置孩子和家庭上下文。
- 建立能力模型。
- 用目标承接阶段方向。
- 用任务和学习计划落地执行。
- 用今日概览完成当天打卡。
- 用图书馆记录阅读。
- 用仪表盘、报告、成就做复盘与激励。

下一阶段不应单纯增加页面，而应围绕三个方向深化：

1. **深度数据闭环**：让任务、阅读、目标、报告能回流到能力模型。
2. **精细化管理**：让图书馆、计划、任务、设置具备更专业的管理能力。
3. **家长减负**：让系统从“记录工具”逐步变成“提醒、建议、复盘助手”。

## 对外部建议的采纳判断

### 采纳，进入近期规划

以下建议方向可取，且与当前产品主线一致。

#### 1. 能力模型编辑

采纳。

原因：

- 当前能力模型仍以静态展示为主。
- 家庭实际使用教材和路线不同，例如 Oxford Discover、RAZ、校内教辅、KET/PET。
- 不允许编辑会导致能力模型很快和真实家庭场景脱节。

落地方式：

- 先支持编辑能力点文案、任务实例、材料要求和阶段描述。
- 暂不支持复杂版本管理。
- 保留系统默认模板，允许恢复默认。

#### 2. 能力模型数据回流

采纳，但不作为最先实现的大功能。

原因：

- 这是产品闭环价值的核心。
- 但它依赖任务、目标、阅读、报告的字段和统计口径，不能直接做“自动点亮”。

落地方式：

- 先建立能力点与任务、目标、阅读记录的关联字段。
- 再建立规则版掌握度。
- 最后再做自动评级和趋势。

#### 3. 报告转任务、报告转目标

采纳。

原因：

- 学习报告如果只能看结果，价值有限。
- 家长真正需要的是“下一步怎么做”。

落地方式：

- 先做规则化建议。
- 建议可以一键生成任务草稿。
- 建议可以关联到已有目标或新目标。
- AI 生成建议后移到未来 2.0 大版本。

#### 4. 阅读精细导入模板

采纳。

原因：

- 当前一书一行的清单只能导入汇总状态。
- 多次阅读同一本书必须是一行一次阅读，否则无法准确记录日期、页码和时长。

落地方式：

- 新增“一次阅读一行”模板。
- 导入前预检区分“书籍清单”和“阅读明细”。
- 阅读明细导入生成多条 `reading_logs`。

#### 5. 图书查重与重复处理

采纳。

原因：

- 图书馆是长期资产，重复书会污染统计。
- ISBN、书名、作者、出版社都可能不完整，需要更明确的合并策略。

落地方式：

- 预检阶段展示疑似重复书。
- 支持选择“合并到已有书 / 新建 / 跳过”。
- 默认优先 ISBN，其次书名 + 作者 + 出版社。

#### 6. 推送通知闭环

采纳。

原因：

- 家庭学习管理需要提醒，不只是记录。
- 钉钉已接入基础配置，应该继续做内容和频率配置。

落地方式：

- 支持每日未完成提醒。
- 支持目标进度异常提醒。
- 支持阅读/运动/习惯分类提醒。
- 支持每个孩子单独配置。

#### 7. 数据导出和备份

采纳。

原因：

- 学习记录是长期资产。
- 数据可导出能显著增强用户信任。

落地方式：

- 先做家庭数据导出。
- 支持 Excel 和 JSON。
- 后续再做定期备份和恢复。

### 采纳，但需要先做基础

#### 1. 多家长协作

方向采纳，但不建议第一个做。

原因：

- 多家长不是简单的邀请按钮。
- 必须先明确家庭成员表、角色、权限、邀请码、退出家庭、移除成员、审计记录。
- 如果先做 UI，后面权限模型容易返工。

建议顺序：

1. 先定义家庭成员和角色模型。
2. 再做邀请家长。
3. 再做权限差异。
4. 最后做协作通知和操作记录。

#### 2. 任务模板库

方向采纳，但先做轻量版。

原因：

- 模板库能降低家长创建任务的门槛。
- 但“科学学习路径”如果设计过重，会变成内容运营项目。

落地方式：

- 先做系统内置模板。
- 支持一键复制为家庭任务。
- 模板按阶段、类型、能力点分类。
- 暂不做复杂课程体系。

#### 3. 学习计划批量操作

方向采纳，但放在计划页稳定后。

原因：

- 批量复制、移动、延期很实用。
- 但需要先保证单个任务移动、删除、同步今日概览足够稳定。

落地方式：

- 批量选择计划项。
- 批量移动到日期。
- 复制本周计划到下周。
- 假期计划模式后移。

### 后移到未来 2.0 大版本

#### 1. AI 辅助分析

后移。

原因：

- AI 建议质量依赖高质量结构化数据。
- 当前应先做规则化建议和数据口径。

2.0 大版本前置条件：

- 能力点、任务、目标、计划、阅读、报告都有稳定数据。
- 页面指标有明确口径。
- 有足够历史数据。

#### 2. 积分商城和复杂成就体系

后移。

原因：

- 当前成就可以先做长期激励展示。
- 积分兑换、奖励商城、防刷规则会显著扩大范围。

2.0 大版本可考虑：

- 成就规则引擎。
- 积分累计。
- 家长自定义奖励。

#### 3. 全自动部署

后移。

原因：

- 当前仍处于频繁修正式环境问题阶段。
- push main 自动上线风险过高。

推荐顺序：

1. 当前：手动推 GitHub + 服务器部署脚本。
2. 下一步：GitHub Actions 手动触发部署。
3. 稳定后：tag 自动部署。

## 版本规划

## 1.7：基础可信与管理补强

定位：把 1.6 的闭环从“页面通了”推进到“关键数据可信、设置可用、导入可控”。

说明：原计划中的 1.6.3 小修不单独发布，合并进入 1.7 一起提交。移动端扫码录入暂不进入 1.7，先把桌面端导入、预检、导出和数据质量做好。

### 1.7-P0

#### 1. 部署与运维标准化

目标：

- 服务器部署只使用 `scripts/deploy-production.sh`。
- 检查只使用 `scripts/check-production.sh`。
- 不再混用 `npm install` 和 `pnpm install`。

验收标准：

- 部署脚本不依赖 `corepack enable`。
- 脚本能完成拉代码、安装、构建、发布、重启、验证。
- 文档说明清楚哪些命令禁止在生产环境执行。

#### 2. 图书馆导入增强

目标：

- 让大规模书单导入更可控。

范围：

- 保留当前导入预检。
- 增加疑似重复书展示。
- 增加书名软匹配相似度评分，书名相同但作者/出版社不一致时允许家长确认或取消匹配。
- 增加“一次阅读一行”的阅读明细导入模板。
- 已读但缺页数/时长的书，标记为“已读完但数据不完整”。

验收标准：

- 《西游记》多次阅读可以导入为多条阅读记录。
- ISBN、出版社、作者、页数、字数能同步到书籍信息。
- 已读状态能同步到对应孩子。
- 导入前能看到预计新增、匹配、跳过、阅读记录和已读同步数量。
- 软匹配行能看到相似度，取消确认后会按新书导入。

#### 3. 设置页补齐

目标：

- 设置页从“展示页”变成可用的管理中心。

范围：

- 家长头像和孩子头像稳定上传。
- 孩子基础信息、学习信息、推送设置、标签配置继续补齐。
- 数据导出入口先做可用。
- 导出 Excel/JSON 时提供数据预检摘要，提示缺 ISBN、缺页数、重复书、任务未关联能力点等问题。
- 错误信息可复制。

验收标准：

- 家长头像和孩子头像刷新后仍能显示。
- 孩子管理能看到当前孩子并可编辑。
- 导出家庭数据可生成文件。
- 图书馆数据可导出 Excel/JSON，导出前能看到数据完整度风险。

### 1.7-P1

#### 4. 任务模板库轻量版

范围：

- 建立内置任务模板。
- 按学习、阅读、运动、习惯分类。
- 支持一键复制为家庭任务。
- 支持标记适用 L1-L5。

#### 5. 学习计划批量操作

范围：

- 批量选择计划项。
- 批量移动日期。
- 复制本周到下周。

#### 6. 推送通知基础闭环

范围：

- 每日任务未完成提醒。
- 每周学习复盘提醒。
- 钉钉推送频率配置。
- 每个孩子单独开关。

## 1.8：能力闭环与数据治理

定位：让能力模型、目标、任务、报告之间形成真实数据回流。

### 1.8-P0

#### 1. 能力模型编辑

范围：

- 编辑 L1-L5 能力点。
- 编辑阶段目标、材料要求、核心任务。
- 编辑时保留系统对标标准作为锚点，例如 RAZ/年级/能力等级参考。
- 支持恢复系统默认模板。
- 支持按孩子或家庭保存自定义模型。

#### 2. 能力点关联字段

范围：

- 任务关联能力点。
- 目标关联能力点。
- 阅读记录关联阅读能力维度。
- 报告输出能力维度结果。
- 从 1.7 起新建任务必须至少关联一个能力点，避免后续数据回流时清洗历史任务。
- 统一任务新建/编辑体验：抽出同一套 `TaskEditor`，将编辑入口收敛为页面级表单，避免任务列表弹窗和任务详情页表单字段不一致。

#### 3. 规则版能力掌握度

范围：

- 根据任务完成次数、目标进度、阅读记录、复盘质量计算初步掌握度。
- 先给出可解释规则，不做黑盒 AI。
- 显示数据来源和更新时间。
- 显示数据置信度；样本不足时提示“数据量不足，评估仅供参考”。

#### 4. 报告行动化

范围：

- 报告识别弱项。
- 报告生成规则化建议。
- 建议一键转任务草稿。
- 建议一键关联目标。

### 1.8-P1

#### 5. 仪表盘诊断化

范围：

- 阅读能力雷达接真实数据。
- 能力趋势解释口径。
- 异常提醒，例如连续三周运动不足、阅读频率下降。

#### 6. 多家长协作基础版

范围：

- 家庭成员模型。
- 邀请家长。
- 角色：主要家长、共同管理者、只读成员。
- 成员列表和移除。

说明：

- 不在 1.8 第一批做复杂权限细分。
- 先保证家庭成员模型正确。

## 1.9：规则化减负与长期激励

定位：在稳定数据基础上，先用规则化方式降低家长维护成本，并补齐长期激励。AI 不进入 1.9，保留到未来 2.0 大版本。

### 1.9-P0

#### 1. 规则化复盘助手

范围：

- 基于任务、目标、阅读、报告生成规则化复盘提示。
- 给出下一步任务草稿和目标调整建议。
- 所有建议展示触发依据。

#### 2. 规则化任务推荐

范围：

- 根据能力短板推荐任务。
- 根据孩子年级、阶段、兴趣推荐阅读任务。
- 推荐必须展示依据。
- 增加家长否定权；点击“不感兴趣”后降低同类推荐权重，为未来 AI 画像做准备。

#### 3. 成就和积分体系

范围：

- 成就来源规则化。
- 积分累计。
- 家长自定义奖励。
- 防刷规则后续扩展。

#### 4. 自动化部署升级

范围：

- GitHub Actions 手动触发部署。
- tag 自动部署。
- 部署日志和失败回滚预案。

## 不建议立即做的事项

### 1. 一上来做复杂 AI

理由：

- 数据还不够稳定。
- 指标口径仍在梳理。
- 容易产生看似聪明但不可解释的建议。

### 2. 一上来做完整多家长权限

理由：

- 权限系统是基础设施，不是单个页面。
- 需要先明确家庭成员模型。

### 3. 一上来做积分商城

理由：

- 内容和运营成本高。
- 当前更需要把真实行为和成就来源打通。

### 4. push main 自动部署

理由：

- 当前生产环境仍需要人工验证。
- 手动触发部署更可控。

## 当前下一步建议

当前 1.7-P0 已进入收尾阶段，发版前重点是验证和发布流程。

### 1.7 发版前检查清单

1. 前后端构建通过。
2. 本地数据库迁移通过，生产发布时执行 `prisma migrate deploy`。
3. 手工验证今日概览、图书馆导入预检、导出、任务能力点必填、孩子头像切换。
4. 修正并验证生产部署脚本，确保不再触发 `corepack enable` 的 `/usr/bin/yarn` 权限问题。
5. 整理本地工作区，只提交 1.7 相关文件，避免把临时备份、导入样例和服务器私钥类文件带入提交。
6. GitHub push 前先 `git pull --rebase` 或 `git pull --ff-only` 处理远端领先问题。
7. 服务器部署后验证 `/api/health`、登录、今日概览、图书馆、任务管理。

### 今日概览后续 Todo

这些问题不阻塞 1.7 发版，建议放到 1.7.1 或 1.8 的体验优化里处理：

1. 学习趋势图改为真实近 7 天数据，而不是当前的静态演示曲线。
2. “AI 今日建议”在真正接入 AI 前改名为“今日建议”，避免用户误解。
3. 底部“学习状态”和“今日学习建议”信息与上方略重复，后续合并为更精简的“今日复盘”。
4. 番茄闹钟目前是前端自由计时工具，不绑定任务、不入库；后续可考虑记录自由专注历史。
5. 任务专注已绑定任务并写入 `focus_minutes`，后续在任务详情页展示专注历史和完成质量。
