# 趣学伴 UI 全面提升方案

更新时间：2026-05-04

## 一、文档目的

本文档用于承接当前 `1.9.1` 阶段的 UI 全面提升工作。

本轮不先讨论后端扩功能，也不先进入数据治理闭环，而是按页面顺序逐页推进 UI 收口。

当前执行策略：

1. 先统一每个页面的职责边界。
2. 再统一按钮、图标、卡片、筛选区和弹窗的层级。
3. 每个页面都按同一套分析框架输出：
   - 现在的问题
   - 我的思考
   - 改进方案
   - 改好后的效果
4. 先从核心页面开始，一个页面一个页面改过去，不并行大面积改动。

## 二、总原则

### 2.1 页面级改造原则

每个页面都必须先回答四个问题：

1. 这个页面第一屏到底要回答什么问题。
2. 页面里哪些是主操作，哪些只是辅助操作。
3. 哪些信息必须第一眼看到，哪些应该后置。
4. 这页的按钮、图标、卡片、弹窗有没有互相抢戏。

### 2.2 统一 UI 判断方法

每个页面统一拆成五层：

1. 页面身份层：这是哪个页面，服务什么任务。
2. 主决策层：这一页最重要的判断和动作。
3. 主工作区：用户真正会停留和操作的区域。
4. 辅助信息层：趋势、解释、建议、说明。
5. 低频层：导出、同步、次级跳转、补充信息。

### 2.3 统一按钮策略

全站按钮只分四级：

- 主操作：默认 `Button`
- 次操作：`Button variant="secondary"`
- 普通操作：`Button variant="outline"`
- 危险操作：`Button variant="destructive"`

### 2.4 统一图标策略

图标只做三件事：

1. 帮助识别页面或模块。
2. 帮助识别状态。
3. 帮助识别动作。

不让图标承担纯装饰任务。

### 2.5 统一卡片策略

卡片只分三类：

- 主卡：本页最重要内容，面积最大，视觉最稳。
- 次卡：补充信息，颜色克制。
- 提醒卡：只在真实有风险时强调。

### 2.6 统一弹窗策略

弹窗统一只保留：

1. 标题
2. 说明
3. 必填项
4. 可选项
5. 底部主次按钮

避免一个弹窗里同时出现太多视觉焦点。

## 三、页面推进顺序

当前按页面顺序推进，不按模块拆分：

1. 今日概览
2. 任务管理
3. 学习计划
4. 目标管理
5. 成长总览
6. 能力模型
7. 阅读与图书馆
8. 数据体检
9. 设置

## 四、页面方案

### 4.1 今日概览页面 UI 改版方案

#### 4.1.1 现在的问题

- 页面主线已经比以前清楚了，但“今日进度 / 下一步行动 / 状态提醒”三块权重太接近，第一眼不够果断。
- 顶栏按钮方向是对的，但页面内部按钮层级还没完全统一。
- `开始下一项` 是这页真正主动作，但下面任务卡里的 `开始`、右侧 `快速操作` 也都很活跃，主次会打架。
- `快速操作` 现在用多组浅彩底按钮，看起来像第二组主功能，不像辅助工具。
- `今日任务` 是主工作区，但下面和右侧还有趋势、建议、快速操作同时出现，任务区统治力还不够。
- 任务卡本身信息是够的，但动作关系还不够清楚。
- 状态标签、按钮、元信息都在右边聚集，局部会有点挤。
- 页面图标使用很多，但不是每个图标都在帮助结构识别。
- `完成任务` 弹窗虽然已经分组了，但视觉上还是有“表单很长、每块都像重点”的问题。

#### 4.1.2 我的思考

`今日概览` 这页只能回答一个问题：

`今天先做什么，任务怎么推进，哪里需要警惕。`

它应该成为“当天执行驾驶舱”，而不是功能集合页。

用户进入这页后，5 秒内必须得到三个结论：

1. 今天整体顺不顺。
2. 下一步该做什么。
3. 任务从哪里开始推进。

#### 4.1.3 改进方案

- 第一屏改成 `8 + 4` 结构，左侧合并“今日完成进度 + 下一步行动”，右侧保留状态提醒。
- `开始下一项` 保留为整页唯一最强按钮。
- `今日任务` 明确成为主工作区，筛选按钮降为工具态。
- 任务卡动作分级：
  - `开始`：`secondary size="sm"`
  - `完成 / 查看`：`outline size="sm"`
- `快速操作` 降权，改成统一工具按钮，不再用多块浅彩底抢主线。
- 弹窗统一收成：
  - 任务摘要
  - 完成结果
  - 反馈记录
  - 1.9 证据
  - 完成日期

#### 4.1.4 改好后的效果

- 第一眼就知道今天整体是否顺利。
- 第二眼就知道应该先点哪个按钮。
- 今日任务真正成为页面核心。
- 快速操作不再抢主线。
- 页面从“信息很多”变成“执行路径很清楚”。

### 4.2 任务管理页面 UI 改版方案

#### 4.2.1 现在的问题

- 顶栏同时有 `导出 / 同步计划 / 任务模板 / 新建任务`，动作都合理，但都在抢第一层注意力。
- 第一屏下面连续出现三层分布、运营指标、任务池结构、需要关注，信息密度高且重要性接近。
- 任务列表其实是主工作区，但被很多前置概览卡稀释了。
- Tab 筛选更像切页面，不完全像切视角。
- 卡片里的 `钉钉` 和 `详情` 同时放在显著位置，容易让“查看详情”变成默认动作。
- 新建、删除、同步计划弹窗的视觉语言还偏旧。

#### 4.2.2 我的思考

这页只能回答一个问题：

`当前孩子的任务池是否合理，我要先改哪个任务，还是新建一个任务。`

它本质上是“任务池管理台”，不是任务数据展览页。

#### 4.2.3 改进方案

- 页面改成四层：
  - 顶栏
  - 任务池总览区
  - 优先处理区
  - 主任务工作区
- 顶栏动作定级：
  - 主操作：`新建任务`
  - 次操作：`同步计划`
  - 普通操作：`任务模板`、`导出`
- 三层分布卡和运营指标合并成“任务池状态面板”。
- “需要关注”升级成待处理队列。
- 任务卡改成任务管理单元，动作重新分级，突出管理和编辑，而不是平铺展示。
- 新建任务弹窗继续强化为“任务配置器”。

#### 4.2.4 改好后的效果

- 页面会更像任务池工作台，而不是统计和列表拼接页。
- 用户会更快找到该处理的任务。
- `新建任务` 会成为真正明确的主动作。
- 任务列表会回到页面核心位置。

### 4.3 学习计划页面 UI 改版方案

#### 4.3.1 现在的问题

- 页面同时承载周切换、筛选、计划摘要、周视图主表、发布计划、推送、导出、任务移动和删除。
- 顶栏动作合理，但 `推送 / 周选择 / 导出 / 发布计划` 都在同一层。
- `本周计划摘要` 和主计划网格关系更像并列信息块，不是服务关系。
- 主计划网格信息量很大，日期头、左列任务、单元格状态、图例都在，阅读压力较高。
- `发布计划` 是核心动作，但目前只是顶栏按钮中的一个。

#### 4.3.2 我的思考

这页只能回答一个问题：

`这一周怎么安排，哪里太满，哪里要调整，我要不要发布这周计划。`

它本质上是“周安排工作台”。

#### 4.3.3 改进方案

- 页面结构改成：
  - 顶栏身份与主动作
  - 周上下文与计划健康摘要
  - 主计划网格
  - 辅助说明和图例
- 顶栏动作定级：
  - 主操作：`发布计划`
  - 次操作：`推送`
  - 普通操作：`周选择`、`导出`
- 周选择强化成“当前周上下文”，不只是普通按钮。
- 摘要区只保留周健康数字和风险提醒。
- 主计划网格成为绝对主角，左列任务信息适度压缩，单元格状态继续做统一。
- `PublishPlanDialog` 定位为“计划发布向导”，一步一步做决策。

#### 4.3.4 改好后的效果

- 用户会先进入计划本身，而不是先被控件带跑。
- 周摘要会更像判断工具。
- 主计划网格会更有统治力。
- `发布计划` 会真正成为核心推进动作。

### 4.4 目标管理页面 UI 改版方案

#### 4.4.1 现在的问题

- 页面同时承载真实目标、能力模型承接、推荐模板、状态分布、下一步动作、数据口径、创建目标、关联任务、复盘。
- 左侧是真实目标主内容，右侧辅助内容很多，容易和主内容争主次。
- 真实目标区域是主工作区，但右侧有很多高可点击内容，注意力被分散。
- `能力模型承接` 和 `推荐目标模板` 都像强入口，容易让用户犹豫从哪建目标。
- 关联任务和目标复盘弹层功能完整，但产品化表达还可以更清楚。

#### 4.4.2 我的思考

这页只能回答一个问题：

`当前孩子最重要的目标是什么，这些目标是否有任务支撑，接下来该新增、调整还是复盘。`

它的主角必须是“真实目标本身”，不是模型入口，也不是模板入口。

#### 4.4.3 改进方案

- 页面结构改成：
  - 顶栏与主动作
  - 目标状态判断区
  - 真实目标主工作区
  - 辅助创建与说明区
- 顶栏主动作统一叫 `创建目标`，而不是直接暴露“从能力点创建”。
- 第一屏状态区更关注：
  - 真实目标数量
  - 已绑定任务数量
  - 需关注目标数量
  - 推进是否稳定
- 左侧真实目标区成为绝对主工作区。
- 右侧辅助区拆成：
  - 创建辅助：能力模型承接、推荐模板
  - 判断辅助：状态分布、下一步动作、数据口径
- 关联任务弹层明确标准路径是“选择已有任务”，生成任务作为兜底方案。
- 目标复盘弹层继续收成“复盘记录器”。

#### 4.4.4 改好后的效果

- 用户第一眼会先关注真实目标，而不是被侧栏入口带跑。
- 创建目标会成为清晰主动作。
- 右侧内容会从第二个页面降为辅助区。
- 目标与任务闭环会更清楚。

### 4.5 成长总览页面 UI 改版方案

#### 4.5.1 现在的问题

- 顶部左侧是 Tab，右侧是时间范围切换，两组强切换控件同时存在，用户先看控件而不是先理解页面。
- 默认页内容很多：总览、指标、结构、趋势、阅读、问题、目标信号、建议都在，信息过满。
- `成长总览` 既像总览页，也像多个专题分析块的拼盘。
- 默认页虽然全面，但用户不一定能马上回答“我现在该先看哪块”。

#### 4.5.2 我的思考

这页只能回答一个问题：

`孩子最近的长期状态怎么样，准备度稳不稳，下一步要补哪类证据或调整哪类方向。`

它必须是“长期判断入口页”，而不是“所有分析块的汇总页”。

#### 4.5.3 改进方案

- 页面结构收成：
  - 顶部分析视角区
  - 总结论区
  - 关键证据区
  - 下一步动作区
- 顶部 Tab 和时间范围都降“控件感”，强化“分析视角”和“分析范围”属性。
- 默认页只保留核心内容：
  - 长期状态结论
  - 四个核心指标
  - 三层结构
  - 完成率趋势
  - 目标准备信号
  - 长期关注问题
  - 家长下一步
- 阅读趋势、时长分布等内容降级，不再和核心判断区平权。
- `家长下一步` 从辅助建议升级成决策卡。

#### 4.5.4 改好后的效果

- 默认页会真正像“长期判断入口页”。
- 用户会先看到结论，再看证据。
- 顶部切换不会再压过页面主体。
- 各个 Tab 的边界会更清楚。

### 4.6 能力模型页面 UI 改版方案

#### 4.6.1 现在的问题

- 页面同时承担模型浏览、阶段切换、能力点编辑、推荐任务、资源推荐、路径说明和模型重置。
- 顶部动作层级不够清楚，`导出报告`、`新增能力点` 和 `编辑模型` 之间业务主次不够匹配。
- 三层卡、阶段路径卡和右侧辅助栏同时存在，主舞台不够突出。
- `AbilityTable` 才是核心工作区，但页面没有完全围绕它服务。
- 右侧辅助栏整体存在感偏高。

#### 4.6.2 我的思考

这页只能回答一个问题：

`当前阶段下，这个孩子在三层准备度模型里的能力结构是什么，哪些能力点需要维护或调整。`

它本质上是“能力模型工作台”。

#### 4.6.3 改进方案

- 页面结构收成：
  - 顶部上下文与主动作
  - 三层切换与当前层概览
  - 能力表格主工作区
  - 辅助解释与延伸内容
- 顶部动作定级：
  - 主操作：`编辑模型`
  - 次级编辑动作：`新增能力点`
  - 普通操作：`阶段切换`、`导出报告`
- 三层卡更像“模型视图切换器”。
- 当前层概览卡只承担：
  - 当前层说明
  - 当前阶段重点
  - 整体进度摘要
- `AbilityTable` 成为绝对主工作区。
- `L1-L5 路径` 降为阶段说明区。
- 右侧整体降为辅助参考区。
- 编辑能力点弹层继续强化为“能力点配置器”。

#### 4.6.4 改好后的效果

- 页面会从“很多信息都相关”变成“主线很清楚”。
- 用户会先理解当前层，再进入表格。
- 表格会回到页面中心。
- 右侧不会再像第二舞台。

### 4.7 阅读与图书馆页面 UI 改版方案

#### 4.7.1 页面定位

本方案覆盖：

- 阅读中心
- 图书馆
- 图书详情

三页作为一条完整链路处理：

`选书 -> 在读 -> 记录 -> 成长档案 -> 修复异常`

#### 4.7.2 现在的问题

- 三页职责大体有了，但边界还没有绝对拉开。
- 阅读中心更像概览页，还不够像在读推进页。
- 图书馆承载内容很多，像功能大本营，主线不够聚焦。
- 图书详情既是档案页，又承担大量操作入口，主线略散。
- 数据质量与阅读修复链路已经打通，但还没完全产品化成顺滑修复体验。

#### 4.7.3 我的思考

这三页必须明确分工：

- 阅读中心：推进页
- 图书馆：管理页
- 图书详情：档案页

#### 4.7.4 改进方案

##### 阅读中心

- 收成“在读推进页”
- 主工作区是在读书卡
- 主动作是更新阅读进度
- 去图书馆选书降成辅助主操作

##### 图书馆

- 收成“书库管理页”
- 顶栏动作定级：
  - 主操作：`添加图书`
  - 次操作：`导入`
  - 普通操作：`模板`、`导出`
- 书库摘要和阅读状态摘要保留，但都为筛选与列表主工作区让路
- 数据质量预检收成“修复入口区”
- 导入预检继续强化成“导入确认台”

##### 图书详情

- 收成“单本书阅读档案页”
- 主动作：`添加阅读记录`
- 成长档案成为页面主角
- 阅读记录明细成为第二主工作区
- 异常修复链路明确为单书页的重要能力

#### 4.7.5 改好后的效果

- 三页职责会完全拉开。
- 阅读中心更有执行感。
- 图书馆更像管理工作台。
- 图书详情更像单书成长档案页。
- 阅读异常修复链路会更顺。

### 4.8 数据体检页面 UI 改版方案

#### 4.8.1 现在的问题

- 页面已经有指标卡、问题卡、规则诊断、优先顺序和快捷入口，方向正确。
- 但主线仍更像“问题总览”，还没有完全变成“修复工作台”。
- `IssueCard` 现在更像摘要卡，不够像修复单元。
- 规则诊断区位置较重，容易削弱“先修数据”的主线。
- “下一步修复顺序”存在，但还不够成为整页第二主锚点。
- 快捷入口还是普通跳转按钮，还不像修复通道。

#### 4.8.2 我的思考

这页只能回答一个问题：

`哪些数据缺口会影响后续判断，我现在应该先修什么，修完后是否有效。`

它必须是一张“修复工作台”。

#### 4.8.3 改进方案

- 页面结构收成：
  - 顶栏与检查上下文
  - 修复状态总览
  - 优先修复顺序
  - 问题修复卡主工作区
  - 规则诊断与辅助入口
- 顶部摘要卡改成修复判断卡，而不只是检查统计。
- `下一步修复顺序` 升级成第二主锚点。
- `IssueCard` 升级成“修复卡”，明确回答：
  - 这是什么问题
  - 影响什么
  - 规模多大
  - 示例是什么
  - 去哪里修
- 规则诊断整体降级成修复后的解释层。
- 底部三个入口继续保留，但重新定义为“修复通道”。
- 强调“修复后回来复查”的闭环体验。

#### 4.8.4 改好后的效果

- 页面会从“问题概览页”升级成“修复工作台”。
- 用户会更快知道先修什么。
- 问题卡会更可执行。
- 规则诊断位置会更准确。
- 页面会更有闭环感。

### 4.9 设置页面 UI 改版方案

#### 4.9.1 现在的问题

- 页面包含领域卡、三大概览、快捷操作和多个配置弹层，信息完整，但主线不够集中。
- 六大领域卡更像首页模块入口。
- `账户概览 / 家庭概览 / 孩子概览` 都很重，连续出现后阅读压力较大。
- 孩子概览承担了大量高频操作，已经接近主工作区。
- 快捷操作和上方领域卡之间有角色重叠。
- `待接入` 信息较多且分散，容易打断页面节奏。

#### 4.9.2 我的思考

这页只能回答一个问题：

`当前设置状态是否健康、是否完整，我下一步应该进入哪个配置域。`

它应是“设置总览工作台”，不是功能目录页。

#### 4.9.3 改进方案

- 页面结构收成：
  - 页面身份与整体状态
  - 核心配置域入口
  - 高频概览区
  - 快捷操作区
  - 待接入和低频区域
- 总览页补出统一页头。
- 六大领域卡收成“配置域导航卡”，减少单卡信息量。
- 领域明确分层：
  - 核心域：孩子管理、账户与安全、家庭与权限
  - 支持域：通知与推送
  - 预留域：数据与隐私、系统偏好
- 三大概览区改成“高频配置摘要区”，并建议把孩子概览提到最高优先级。
- 快捷操作改成补充型快速入口，不再和主设置域平权。
- 待接入内容尽量集中收纳，不分散在主工作区。
- 资料编辑、钉钉配置、学习设置弹层统一成“设置配置器”。

#### 4.9.4 改好后的效果

- 页面主线会更清楚。
- 设置页会更像工作台，而不是功能目录页。
- 孩子配置会成为真正核心。
- 待接入信息不会再打断主线。
- 弹层会更统一。
