# 趣学伴 1.9.2 三层准备度内核定型迭代计划

更新时间：2026-05-05

## 一、版本定位

`1.9.2` 只做一件事：

```text
先定三层准备度内核，再让目标管理承接它。
```

本版本不继续扩数据体检，不做 AI，不做报告，不做新的阅读链路，不做移动端专项。

上一阶段已经完成三层准备度的信息架构迁移，但当前三层模型仍偏“分类框架”。如果三层准备度没有先定清楚，目标管理、任务推荐、成长总览、报告和 AI 都会缺少约束。

本版本的第一优先级不是先改目标管理页面，而是把三层准备度作为系统内核定型：

- 三公基础模板怎么定义。
- 交付层、认知层、稳定性层分别包含哪些能力点。
- 每个能力点看什么观察指标。
- 每个能力点对应哪些推荐任务。
- 每个能力点的数据来源是什么。
- 后续如何承接基础线、优秀线和三公目标线。

目标管理是三层准备度的第一应用场景。目标管理必须服从三层模型，用来回答：

- 当前目标到底服务哪个准备方向。
- 三层准备度分别缺什么。
- 哪些任务在支撑目标。
- 下一步应该补能力、补证据、补任务，还是调整目标。

因此，本版本的核心目标是把三层准备度从“模型展示页”推进到“系统内核页”，再把目标管理从“目标列表页”推进到“模型应用工作台”。

## 二、范围边界

### 2.1 本版本做

- 三层准备度模型结构收口。
- 三公基础模板首版。
- 目标方向配置。
- 目标与三层能力点的关系强化。
- 目标卡片和目标详情重构。
- 目标准备状态判断。
- 目标支撑任务关系梳理。
- 目标缺口提示和下一步动作。

### 2.1.1 发版前稳定性修正

本版本允许包含一个非产品功能修正：

- 修复生产环境 API 限流过严导致的登录失败、孩子数据加载失败和健康检查 429。

问题背景：

- 生产环境首页会并发请求今日概览、计划、成长和孩子数据。
- 当前全局限流会把 `/api/health`、`/api/version`、`/api/login`、`/api/children` 等接口一起计入。
- 一旦达到 `100/15min`，用户会看到登录失败、孩子数据刷不出来，运维检查也会返回 `429 Too Many Requests`。

修复要求：

- `/api/health` 和 `/api/version` 不进入普通业务限流。
- 普通业务 API 的生产限流提高到适合单页应用并发请求的水位。
- 登录接口保留独立较严格限流。
- 登录限流只作用于 `/api/login`、`/api/register` 等认证入口，不影响 `/api/children` 和其他已登录业务接口。
- 发版后 `scripts/check-production.sh` 不应因为限流返回 429。

### 2.2 本版本不做

- 不做数据体检完整修复闭环。
- 不做目标校录取概率。
- 不做上实、上外、浦外适配分。
- 不做 AI 建议。
- 不做自动生成报告。
- 不做复杂后端诊断引擎。
- 不做移动端专项改版。

### 2.3 允许预留但不展示强结论

可以在数据结构和页面文案里预留：

- 三公基础模板。
- 上实方向。
- 上外方向。
- 浦外方向。
- 目标权重配置。
- 目标水位。

但本版本只展示“准备信号”和“证据缺口”，不展示录取概率或学校适配分。

## 三、产品目标

### 3.1 用户问题

家长进入三层准备度和目标管理页面后，应该能回答四个问题：

1. 三公准备到底由哪些能力构成。
2. 当前孩子的目标应该落在哪些层和能力点上。
3. 现有任务是否真的支撑这些目标。
4. 下一步应该先补模型、补目标、补任务，还是先稳定节奏。

### 3.2 产品形态

三层准备度页成为系统内核页：

```text
三公基础模板 -> 三层结构 -> 能力点 -> 观察指标 -> 推荐任务 -> 数据来源
```

目标管理页成为模型应用工作台：

```text
目标方向 -> 三层能力点 -> 目标列表 -> 支撑任务 -> 下一步动作
```

两页关系必须清楚：

- 三层准备度回答“模型如何定义，能力点如何维护”。
- 目标管理回答“当前要推进什么目标，下一步怎么做”。

## 四、三公目标下的三层定义

### 4.1 交付层

交付层回答：

```text
现在够不够强。
```

三公基础模板的交付层包含：

- 英语能力：原版阅读、听说表达、词汇句法、结构化输出。
- 数理逻辑：数学思维、新规则题、空间想象、稳定解题。
- 大语文：中文阅读、古文国学、百科文史、表达写作。
- 项目成果：作品、活动、荣誉、可展示材料。
- 临场应变：面谈表达、现场推理、追问应对、综合沟通。

### 4.2 认知层

认知层回答：

```text
为什么能变强。
```

三公基础模板的认知层包含：

- 信息处理速度。
- 规则内化。
- 逻辑深度。
- 批判思维。
- 迁移应用。

本版本不做自动认知评分，但要让目标和任务能明确绑定认知能力点。

### 4.3 稳定性层

稳定性层回答：

```text
能不能持续变强。
```

三公基础模板的稳定性层包含：

- 睡眠作息。
- 运动健康。
- 情绪恢复。
- 执行完成率。
- 延期率。
- 复盘频率。

本版本只做目标准备判断中的稳定性提示，不做完整过载诊断。

## 五、核心改造

### 5.1 三层准备度页

#### 现在的问题

- 页面已经能展示三层结构，但仍偏模型说明。
- 三公基础模板、水位、观察指标和推荐任务之间关系不够清楚。
- 能力点和目标管理之间的跳转关系不够强。
- 旧“能力模型”和新“三层准备度”的边界还需要进一步收口。

#### 改造目标

把三层准备度页改成“模型配置和能力点维护页”。

页面结构：

1. 顶栏：三层准备度，显示当前孩子阶段和当前模板。
2. 模板区：三公基础模板，预留上实/上外/浦外方向。
3. 三层概览：交付层、认知层、稳定性层。
4. 当前层能力点表：能力点、观察指标、推荐任务、关联目标数。
5. 右侧辅助区：三层解释、目标管理入口、模板说明。

#### 必须完成

- 三层页标题统一为“三层准备度”。
- 当前模板默认显示“三公基础模板”。
- 每个能力点展示：
  - 所属层级。
  - 能力点名称。
  - 观察指标。
  - 推荐任务。
  - 已关联目标数。
  - 已关联任务数。
- 支持从能力点进入目标创建。
- 支持从能力点进入任务创建。
- 旧能力模型文案降级，不再和三层口径平权。

### 5.2 目标管理页

#### 现在的问题

- 目标列表已经存在，但目标主线不够强。
- 右侧辅助入口较多，容易抢主工作区注意力。
- 目标与三层能力点、任务支撑、准备缺口之间关系不够直接。
- 创建目标入口仍偏“从能力点创建”，不是完整目标配置器。

#### 改造目标

把目标管理页改成“目标准备工作台”。

页面结构：

1. 顶栏：目标管理，主操作为“创建目标”。
2. 目标方向区：当前目标方向、准备周期、三层覆盖情况。
3. 目标准备判断区：目标数量、支撑任务、缺口、稳定性风险。
4. 真实目标主工作区：目标卡片列表。
5. 辅助区：能力点创建、推荐模板、状态分布、数据口径。

#### 目标卡片字段顺序

目标卡片必须优先展示：

1. 目标名称。
2. 目标方向或目标类型。
3. 所属三层能力点。
4. 当前进度。
5. 成功标准。
6. 支撑任务数量。
7. 当前缺口。
8. 下一步动作。

低频字段后置：

- 复盘历史。
- 详细描述。
- 推荐说明。
- 数据口径。

#### 必须完成

- 顶栏主按钮统一为“创建目标”。
- 创建目标弹窗升级为“目标配置器”。
- 目标必须能配置：
  - 目标名称。
  - 目标方向。
  - 目标周期。
  - 所属层级。
  - 能力点。
  - 成功标准。
  - 支撑任务。
  - 复盘节奏。
- 目标卡片展示“三层缺口”和“支撑任务”。
- 目标缺任务时给出“关联任务”动作。
- 目标缺成功标准时给出“补完成标准”动作。
- 目标缺能力点时给出“补能力点”动作。
- 关联任务弹窗标准路径为“选择已有任务”，生成任务只作为兜底。

### 5.3 目标方向

本版本新增目标方向口径，但只做轻量配置。

默认方向：

- 三公基础。
- 英语强化。
- 数理逻辑。
- 阅读表达。
- 稳定执行。

预留方向：

- 上实方向。
- 上外方向。
- 浦外方向。

目标方向不等于录取判断。页面文案必须避免：

- 适配分。
- 录取概率。
- 排名。
- 冲刺成功率。

推荐表达：

- 目标方向。
- 准备信号。
- 证据缺口。
- 支撑任务。
- 下一步动作。

## 六、数据与规则口径

### 6.1 本版本使用现有数据

优先使用现有字段和 `tags`/`settings` 能承载的结构，不做大规模 schema 迁移。

可使用现有数据：

- 任务 `tags.abilityCategory`。
- 任务 `tags.abilityPoint`。
- 任务 `tags.linkedGoal`。
- 任务 `scheduleRule`。
- 任务 `trackingType`。
- 目标设置。
- 完成记录。
- 打卡 metadata。

### 6.2 目标准备判断规则

首版只做轻量规则，不做复杂评分。

目标状态可分为：

- `待建立`：目标缺关键字段。
- `待支撑`：目标存在，但缺支撑任务。
- `推进中`：目标有支撑任务和成功标准。
- `需关注`：目标有缺口或相关任务执行不稳定。
- `可复盘`：目标已经有阶段性记录，可以复盘。

### 6.3 三层覆盖判断

目标管理页只展示覆盖情况，不输出强结论：

- 交付层目标数量。
- 认知层目标数量。
- 稳定性层目标数量。
- 已绑定任务的目标数量。
- 缺成功标准的目标数量。
- 缺复盘记录的目标数量。

## 七、验收标准

### 7.1 产品验收

- 家长进入目标管理页，10 秒内能看到当前目标方向、主要缺口和下一步动作。
- 家长能从目标卡片直接知道该目标服务哪个三层能力点。
- 家长能看出哪些目标没有任务支撑。
- 家长能从三层准备度页进入目标创建或任务创建。
- 三层准备度页不再像静态说明页，而是能服务目标配置。

### 7.2 三公口径验收

- 页面默认支持“三公基础模板”。
- 交付层包含英语、数理、大语文、项目成果、临场应变。
- 认知层包含信息处理速度、规则内化、逻辑深度、批判思维、迁移应用。
- 稳定性层包含睡眠作息、运动健康、情绪恢复、执行完成率、延期率、复盘频率。
- 页面不输出录取概率、排名和目标校适配分。

### 7.3 工程验收

- `pnpm --filter frontend lint` 通过。
- `pnpm --filter frontend build` 通过。
- `pnpm --filter quxueban-backend lint` 通过。
- `pnpm --filter quxueban-backend build` 通过。
- `pnpm run check:versions` 通过。
- smoke 测试通过。
- 生产环境 `/api/health`、`/api/version`、`/api/login`、`/api/children` 不因正常页面加载触发 429。
- `scripts/check-production.sh` 连续执行不应把生产 API 打入不可登录状态。
- 桌面端截图检查覆盖：
  - 目标管理。
  - 三层准备度。
  - 目标创建弹窗。
  - 关联任务弹窗。

## 八、实施顺序

### Step 1：三层准备度内核定稿

- 固定三公基础模板。
- 固定三层能力点列表。
- 固定每个能力点的观察指标。
- 固定每个能力点的推荐任务。
- 固定每个能力点的数据来源。
- 固定目标方向列表。
- 固定目标状态规则。

### Step 1.5：生产限流护栏修复

- 调整后端限流挂载顺序。
- 健康检查和版本接口避开业务限流。
- 普通业务 API 限流水位提高。
- 登录接口保留独立限流。
- 验证生产登录和孩子列表接口不再因正常页面加载触发 429。

### Step 2：三层准备度页改造

- 页面结构改成模板、三层、能力点、辅助说明。
- 能力点增加关联目标和关联任务信息。
- 增加从能力点创建目标和任务的入口。

### Step 3：目标管理承接三层模型

- 顶部改成目标方向和准备判断。
- 目标卡片重排。
- 辅助入口降权。
- 创建目标弹窗升级为目标配置器。

### Step 4：目标与任务支撑闭环

- 关联任务弹窗标准化。
- 缺任务、缺能力点、缺成功标准的目标给出动作。
- 目标保存后刷新目标、任务和成长相关缓存。

### Step 5：验收与发版

- 跑 lint/build/version/smoke。
- 补桌面截图记录。
- 写发布记录。

## 九、后续版本衔接

本版本完成后，后续版本再继续：

- `1.9.3`：数据治理修复闭环。
- `1.9.4`：规则诊断增强。
- `1.9.5`：目标校权重和准备度信号增强。
- `2.0`：AI 目标建议和任务推荐。

注意：如果 `1.9.2` 没有先把三层准备度内核定清楚，不建议继续扩目标校权重、规则诊断或 AI。
